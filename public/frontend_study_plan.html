面試準備計畫

從「會寫功能」到「會設計系統」，建立穩定輸出能力。

主線 A 基本盤 重點是穩定講出思路，不是刷題數量
A1 資料結構與演算法 + 刷題 20%
  • 不求看完 CS50，看得懂、說得出口就停
  • Easy 全通 + 常見 Medium，目標 30–35 題
  • 每題要能回答：題目在問什麼？為什麼這樣解？時間複雜度大概是多少？
刷題重點不是解出來，是練習說清楚。
每題結束後用英文說一遍：My idea is to use a hash map so we can achieve O(n) time complexity.
Array HashMap Two Pointers Sliding Window BFS / DFS Stack DP（基本款）
A2 技術問答 10%
  • React rendering model（reconciliation、fiber、concurrent mode）
  • State lifting vs global state，決策點在哪
  • Next.js data flow（SSR / CSR / hydration / ISR）
  • React Query vs Redux 的取捨
  • TypeScript generic、utility types、型別設計思維 補充
  • 效能優化：Core Web Vitals、lazy loading、code splitting、memo/useMemo/useCallback 使用時機 補充
  • 前端測試：unit / integration / e2e 的分層、Testing Library 使用思路 補充
  • 打包工具：Vite 原理、tree-shaking、HMR、build optimization 補充
A3 CS 基礎 10%
  • 記憶體基本概念：stack / heap
  • Reference vs value，對 JS/TS 的實際影響
  • Big-O 直覺，不用背公式
  • Array / Object / Map / Set 的使用取捨
  • 瀏覽器原理：Event loop、call stack、microtask / macrotask、reflow / repaint 補充
  • 網路基礎：HTTP/1.1 vs HTTP/2、CORS、cookie / session / JWT 機制 補充
  • 可訪問性（Accessibility）：WCAG 基本概念、語意化 HTML、ARIA 屬性 補充
主線 B 前端架構成熟度 從「會寫功能」變成「會設計專案」
B1 狀態分層訓練

每週問自己一次：

  • 哪些 state 是 UI state？（modal open、tab active）
  • 哪些是 domain state？（user data、event list）
  • 哪些應該 derived？（filtered list、computed total）
  • 哪些根本不應該存在？（可直接從 props / context 拿）
B2 分層設計能力

練習講出這種結構，並說清楚每層的職責：

/features /event eventSlice.ts # state 定義、reducer eventService.ts # API call、資料轉換 eventSelectors.ts # derived state、memo /components # 純 UI，不知道 API schema /hooks # 業務邏輯 + UI 的橋接 /utils # 純函數，無副作用
  • API 資料轉換在哪層？（service 層）
  • Error handling 在哪層？（service + slice）
  • Cache 怎麼做？（React Query / RTK Query / 手動 TTL）
  • Loading state 怎麼設計？（slice status field / React Query isLoading）
B3 邊界設計
🧱 Component 不應該知道 API schema，只接收乾淨的 domain model
🧱 Redux / global store 不應該知道 UI 結構（哪個 modal 開著）
🧱 UI 不應該直接做資料轉換（formatting、mapping 放 util / selector）
🧱 系統設計延伸：能回答「設計一個可擴展的表單系統 / 元件庫」等架構題 補充
這是區分中階與高階工程師的關鍵。
面試官問「你怎麼設計這個功能」時，你在回答的其實是邊界認知。
B4 重構練習

每週一次：挑一段自己過去寫的程式，問：

  • 這段耦合在哪？
  • 可否抽 service？（API 邏輯不應混在 component）
  • 可否抽 selector？（derived state 不應每次重算）
  • 可否去除 redundant state？
這種練習的成長速度是刷題的 5 倍。
因為你在練的是工程判斷力，不是解題技巧。
B5 效能優化實作 補充
  • React DevTools Profiler：找出不必要的 re-render
  • bundle analyzer：找出過大的依賴
  • Lighthouse / Web Vitals：LCP、CLS、FID 的優化思路
  • 能說清楚：為什麼這裡要 memo？為什麼不要？
面試常考：你遇過什麼效能問題，怎麼診斷、怎麼解？
B6 CI/CD 基礎認知 補充
  • 前端如何接入 pipeline（lint → test → build → deploy）
  • 自動化測試如何在 CI 跑
  • 環境變數管理（.env、staging / production 分離）
  • 不用到 DevOps 程度，但要能說清楚流程
主線 C 輸出與表達 建立「看得懂題目，也講得出思路」的能力
C1 架構口述練習

每週至少一次回答：「如果讓你設計一個中型 React 專案，你會怎麼設計資料流？」

  • state 放在哪裡（local / global / server state）
  • API flow（fetch → transform → store → render）
  • component 分層（feature / shared / ui）
  • loading / error handling 策略
  • cache strategy（React Query TTL / invalidation / optimistic update）
目標：可以連續講 5 分鐘不亂。
從 3 分鐘開始練，每次錄音或寫下來，找自己說不清楚的點。
C2 演算法思路表達

刷題後不要只寫答案，練習說出：

  • 題目在問什麼（用自己的話說）
  • 為什麼選這個解法（不是只說「我想到用 hash map」）
  • 有沒有其他解法？各自的 trade-off？
  • 時間複雜度 + 空間複雜度
練習開場句：
"My idea is to use a hash map to store visited values so we can achieve O(n) time complexity."

目標：2–3 分鐘講清楚一道題的完整思路。
C3 技術決策說明

練習這類問題的回答：

  • 為什麼用 Redux？什麼情況不用 global state？
  • React Query vs Redux，怎麼選？
  • 為什麼要做 selector？不做的後果是什麼？
  • 為什麼用 TypeScript？如何設計型別邊界？ 補充
  • 為什麼要寫測試？哪些先寫、哪些不寫？ 補充
回答框架：問題 → 選擇 → 原因 → trade-off

例："We chose Redux because multiple pages share the same event state, and it avoids prop drilling. The trade-off is boilerplate — for simpler cases we'd use React Query instead."
C4 專案經驗敘事

準備 3 個故事，每個 2–3 分鐘：

🐛 最難的 Bug
背景 → 現象 → 排查過程 → 根因 → 解法 → 事後預防
🔨 最大的重構
為什麼重構 → 哪裡耦合 → 怎麼拆 → 結果（效能、維護性、測試覆蓋）
💡 最有價值的設計決策
背景 → 考慮過的選項 → 為什麼選這個 → trade-off → 結果
加分項：故事裡有數字更好。
「效能提升了 40%」比「速度變快了」有說服力。
C5 英文口說練習 補充整合
  • 目標：可工作等級（technical discussion、code review、meeting）
  • 不需要口音完美，需要能精準表達技術思路
  • 每次刷完題後用英文說一遍思路（2 分鐘錄音）
  • 每週一次架構口述，用英文說（不需要全英，但要能切換）
重點句型練習：
"The reason I chose X is..." / "The trade-off here is..." / "This approach has O(n) time complexity because..."
主線 D 工程品質意識 新增主線 讓面試官感受到你不只是「功能實現者」
D1 前端測試策略
  • 測試金字塔：unit（純函數、selector）→ integration（component + hook）→ e2e（關鍵路徑）
  • React Testing Library：測行為，不測實作
  • 什麼值得測？什麼不用測？能說清楚判斷依據
  • MSW（Mock Service Worker）：前端 API mock 最佳實踐
D2 打包工具理解
  • Vite：ESM、HMR、Rollup build 原理
  • Tree-shaking：為什麼有些東西 shake 不掉
  • Code splitting：dynamic import、route-based split
  • Bundle analyzer：怎麼診斷包太大的問題
D3 可訪問性 Accessibility
  • WCAG 2.1 基本概念（A / AA 等級）
  • 語意化 HTML：用對標籤比加 ARIA 更重要
  • 鍵盤導航、focus management
